# Unity 项目结构小记

短周期项目里，最容易失控的不是代码量，而是职责边界。一个功能刚开始可能只有几行脚本，但如果数据、表现、输入和状态全写在一起，后面改起来会非常吃力。

我目前更倾向把项目拆成几个清晰部分：

- 玩法逻辑：规则、状态、胜负条件和核心计算
- 表现层：动画、特效、音效、UI 展示
- 数据配置：关卡、数值、武器、技能和可调参数
- 工具流程：编辑器配置、调试入口和内容生产辅助

这个拆分不需要一开始就做得很重。小项目里只要保持“能看懂、能替换、能调试”，就已经能减少很多后期返工。

```csharp
public interface IGameSystem
{
    void Initialize();
    void Tick(float deltaTime);
}
```

真正重要的是团队里的其他人也能快速理解：这个功能在哪里改，哪里只是表现，哪里会影响核心规则。
